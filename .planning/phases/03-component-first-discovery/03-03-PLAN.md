<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User search input -> discovery ranking | Untrusted input determines filtering, scoring, and displayed match reasons. |
| Mirror metadata -> profile view model | Third-party/public registry facts become inspectable user-facing profile content. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Tampering | `discovery.ts` | mitigate | Normalize search terms and return data-only match reasons; rendering/escaping stays in UI layer. |
| T-03-03-02 | Spoofing | `registryProfile.ts` | mitigate | Separate `Official shadcn facts` from `Registry Atlas enrichment` and use neutral coverage labels. |
| T-03-03-03 | Repudiation | candidate ranking | mitigate | Include `matchedField`, `matchReasons`, and coverage label so users can see why a result appeared. |
| T-03-03-04 | Denial of Service | search over local data | accept | Dataset is a small static mirror; pure array scans are bounded by local registry count. |
| T-03-03-SC | Tampering | npm installs | accept | No new package-manager installs are planned for this phase plan. |
</threat_model>
