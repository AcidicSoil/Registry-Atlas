## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon/text gaps, chip gaps, table cell micro-spacing |
| sm | 8px | Compact row padding, list gaps, card gaps |
| md | 16px | Default body padding, app-inner gaps, header-to-content spacing |
| lg | 24px | Desktop section padding and profile section separation |
| xl | 32px | Major content group gaps and wide profile split layouts |
| 2xl | 48px | Empty-state breathing room and wide-screen content breaks |
| 3xl | 64px | Page-level spacing only; do not introduce hero spacing in this phase |

Exceptions: Existing radii may remain at 12px, 13px, 14px, 16px, 20px, and 28px where already defined. Phase 3 interactive rows must use a minimum 36px desktop target and 44px mobile/touch target. Status chips must use 4px 8px internal padding to match the approved spacing scale.

## Typography

Use only these four sizes for new Phase 3 UI. Reuse existing smaller inherited styles only when touching unchanged components.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 12px | 400 | 1.5 |
| Label | 11px | 600 | 1.35 |
| Heading | 14px | 600 | 1.2 |
| Display | 17px | 600 | 1.2 |

Typography rules:
- Body text covers descriptions, result summaries, profile facts, empty-state body copy, and help copy.
- Labels cover chips, metadata, status, provenance, table headings, and row eyebrow text.
- Headings cover content titles, result section titles, selected component names, and profile section titles.
- Display is reserved for the persistent `Registry Atlas` brand only; do not add hero-scale headings.
- Letter spacing remains `0` for body and chip-compact text. Uppercase operational labels may use the existing `0.14em` to `0.16em` tracking only where already used by tabs, titles, and table headings.
- Do not introduce additional font weights beyond 400 and 600 for Phase 3 UI. Existing logo weight may remain unchanged.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#05060a` | App background, content wells, default result surface, matrix body |
| Secondary (30%) | `#101320` | Sidebar, secondary cards/rows, profile panels, table header surfaces |
| Accent (10%) | `#ffd95e` | Active tab/pill, verified item status, primary metric values, selected result outline |
| Destructive | not used | Destructive actions only; Phase 3 has no destructive actions |

Accent reserved for: active primary navigation state, selected filter/result state, verified item status, primary count emphasis, and the "Open item route" link only when the item route is validated. Do not use accent for unverified coverage, warnings, generic hover decoration, or unavailable routes.

Secondary semantic color: `#65d4ff` is reserved for neutral review/status affordances: official source links, mirror review state, warnings, inferred coverage, profile/source links, and non-primary inline links. It must not imply approval or safety.

Status color contract:

| Status | Label | Color Treatment |
|--------|-------|-----------------|
| verified | Verified item | `#ffd95e` text/border on dark surface |
| inferred | Inferred from Atlas tags | `#65d4ff` text/border on dark surface |
| partial | Partial catalog | `#65d4ff` dashed border, muted dark fill |
| unavailable | Catalog unavailable | `#9ea0b6` text, dashed border |
| unverified | Unverified coverage | `#6a6d84` text, low-emphasis border |

Do not add decorative gradient/orb redesigns. Existing dark radial backgrounds may remain, but new discovery rows, profile sections, and status cells must be quiet, dense, and scan-first.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | View profile |
| Valid item route link | Open item route |
| Unavailable item route label | Item route unavailable |
| Unverified route label | Catalog not verified |
| Empty state heading | No verified item matches yet |
| Empty state body | Showing unverified registry matches when available. Refine the search or inspect a registry profile for source and coverage context. |
| No matches state | No registry or component matches found. Try a component name, capability, namespace, or focus tag. |
| Partial-data state | Some registries do not expose a verified item catalog. Results may include inferred or unverified coverage. |
| Error state | Something went wrong while rendering this view. Clear the search or reload the page. |
| Destructive confirmation | none; Phase 3 has no destructive actions |

Required labels:
- Use `Verified item`, `Inferred from Atlas tags`, `Partial catalog`, `Catalog unavailable`, and `Unverified coverage` exactly for coverage statuses.
- Use `Official shadcn facts` for upstream namespace, homepage, source, registry URL template, source URL, and sync metadata.
- Use `Registry Atlas enrichment` for focus tags, component tags, aliases, confidence, notes, and inferred coverage.
- Use `Why this matched` for profile context opened from a search result.
- Use `Warnings need review`, not "unsafe", "approved", "secure", "trusted", or "certified".

Out-of-scope copy:
- Do not add `Copy install`, `Copy view`, `Add to queue`, `Batch install`, or URL sharing copy in Phase 3.
- Do not show disabled Phase 4 buttons. If future-action layout preparation is needed, reserve data fields in view models, not visible controls.

## Phase 3 Interaction Contract

### Default Surface

- The first screen must be the discovery tool, not a landing page.
- The default workflow is component-first search results. Focus, Component, and Matrix remain available as secondary comparison modes.
- The existing single search input remains the primary input. Update the placeholder to `Search components, items, registries, or aliases`.
- Default content with an empty query must show useful component-first discovery scaffolding: high-signal component/capability candidates, current mirror counts, and status distribution. It must not be a marketing hero.
- Search must update results in place without navigation, server calls, or URL state in Phase 3.

### Component-First Result Rows

Each result row must show, in this order:

| Zone | Required Content |
|------|------------------|
| Primary label | Matched component/capability label, or item name when known |
| Registry context | Registry namespace/name and short description |
| Item context | Item slug when known; otherwise `Item slug unknown` |
| Route context | `Open item route` link only for validated resolved item URLs; otherwise `Item route unavailable` or `Catalog not verified` |
| Status | Coverage status chip and confidence/provenance chip |
| Match reason | One short `Why this matched` phrase such as `Exact item match`, `Atlas tag match`, `Alias match`, or `Description match` |
| Affordances | `View profile` and secondary source/homepage links |

Row layout:
- Use compact comparison rows for desktop, not large cards.
- Use a single-column stacked row on mobile.
- Keep row height stable across statuses; route/status chips must not shift adjacent columns.
- Use `button` only for in-app selection/profile affordances and `a` only for external links.
- External links open in a new tab with the existing safe-link behavior.

### Validated Item Routes

- A route may be shown as `Open item route` only when namespace, registry URL template, item slug, and resolved URL are valid.
- A registry homepage link must never be presented as a component/item route.
- When only registry-level metadata matches, show the registry candidate lower in the list with `Unverified coverage` and `Catalog not verified`.
- If a route cannot be resolved, keep the row visible and explain the missing piece using neutral copy.

### Search Ranking And Degradation

- Exact validated item/component matches rank first.
- Exact component tag matches rank second.
- Alias and focus matches rank third.
- Namespace, description, and metadata matches rank after direct component signals.
- Verified, inferred, partial, unavailable, and unverified statuses are tie-breakers in that order.
- Do not hide unverified records. Sort them lower and label them clearly.
- If no verified item matches exist, show the empty/partial-data copy above unverified or inferred registry matches.

### Registry Profile

Profile view must be inspectable from every result row and from secondary registry surfaces. It must include:

| Section | Required Fields |
|---------|-----------------|
| Header | Registry namespace/name, short description, coverage status, confidence, warning count |
| Official shadcn facts | Homepage/source link, registry URL template, source URL, last sync, official directory count context |
| Registry Atlas enrichment | Focus tags, component tags, aliases, notes, confidence, Atlas coverage status |
| Item discovery status | Known item candidates, validated item routes, unavailable/unverified catalog state |
| Why this matched | Match fields and reasons when opened from search |
| Source actions | Homepage/source/item route links only; no copy commands or queue controls |

Profile layout:
- Use un-nested full-width sections inside the existing content panel; do not put cards inside cards.
- Distinguish official facts and Atlas enrichment with section headings and provenance chips, not approval badges.
- Long registry URL templates must wrap or truncate safely with accessible full text, without horizontal page overflow.

### Secondary Views

- Focus view remains a registry-cluster browser and must add coverage distribution/status hints when group data is available.
- Component view becomes a component index/facet supporting the primary discovery flow. Selecting a component must align with component-first results rather than only listing registry homepages.
- Matrix view must distinguish `verified`, `inferred`, `partial`, `unavailable`, and `unverified` when data exists. Unknown/unverified cells must not look like confirmed absence.
- Existing mirror status remains visible in the content header and keeps neutral Phase 2 language.
